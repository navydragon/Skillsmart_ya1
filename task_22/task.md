Вернёмся к архитектуре на монадах состояний (монадический поток управления) с контролем ошибок. В ней всё прекрасно за исключением того, что если робот столкнулся с препятствием, последующие команды move продолжат бесполезные попытки его двигать, а отсутствие воды всё равно не отменит попытки выполнения команд смены состояния чистильщика.

Более правильный архитектурный подход на основе возможностей (capabilities) должен предоставлять следующее:
- после каждой операции возвращается новый набор доступных функций;
- функции могут быть недоступны в определенных ситуациях;
- состояние скрыто внутри замыкания.

То есть мы хотим сделать наш API ещё более безопасным, так как клиентский код не сможет выполнить недопустимые операции -- они просто будут отсутствовать в наборе доступных функций.
