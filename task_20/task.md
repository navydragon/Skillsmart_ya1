21. Абстрактный тип данных
Давайте вернёмся к первым заданиям.

В объектно-ориентированной реализации детали внутренних компонентов хорошо инкапсулированы, и доступ осуществляется только через методы. Главный недостаток класса -- он (его экземпляр) изменяемый.

В функциональной реализации состояние неизменяемое, но внутренние компоненты состояния общедоступны.

Подумайте как сделать реализацию, которая сочетает лучшие черты ООП и ФП.

От ОО хорошая инкапсуляция и сокрытие деталей реализации.
От функциональщины неизменяемость и чистые функции.
Дополнительно получим строгий контроль типов и явное разделение публичного API и внутренней реализации.

---

## Вариант: opaque ADT + чистые функции

Идея: наружу отдаём **абстракцию `Robot`**, у которой **нет публичных полей**. Пользователь может работать с роботом только через набор **чистых функций**. Внутреннее представление состояния спрятано в модуле и может меняться без изменения пользовательского кода.

### Публичный контракт (API)

Модуль: `task_20/robot_adt.py`

- **Типы и константы**
  - `Robot` — непрозрачный тип (opaque): нельзя полагаться на структуру/поля
  - `CleaningMode` — режим уборки как строковый литерал (`"water" | "soap" | "brush"`); в `create(...)` по-прежнему можно передать `mode` как `WATER`/`SOAP`/`BRUSH` (`int`)
  - `WATER`, `SOAP`, `BRUSH` — константы режимов

- **Создание и “чтение” (без доступа к полям)**
  - `create(...) -> Robot`
  - `position(robot) -> tuple[float, float]`
  - `angle(robot) -> float`
  - `mode(robot) -> CleaningMode` (строковый литерал `"water" | "soap" | "brush"`)

- **Чистые операции (не мутируют вход, возвращают новый `Robot`)**
  - `move(robot, distance) -> Robot`
  - `turn(robot, angle_deg) -> Robot`
  - `set_mode(robot, new_mode) -> Robot`

- **Исполнение программы (команды — часть публичного API)**
  - `run(commands, robot) -> tuple[Robot, list[Any]]`
    - `commands` — последовательность объектов `Start`, `Move`, `Turn`, `SetMode`, `Stop` из `robot_adt` (у каждого есть `execute(transfer, state)`; внутри вызывается `pure_robot`)
    - возвращает нового робота и лог, собранный через `transfer`

### Как достигается “лучшее из ООП и ФП”

- **Инкапсуляция (ООП)**: состояние хранится во внутренних типах (приватная реализация). Пользователь не может “залезть” в поля и нарушить инварианты.
- **Неизменяемость (ФП)**: операции `move/turn/set_mode/run` возвращают новый `Robot`, не изменяя исходный.
- **Чистые функции**: вход → выход без скрытых побочных эффектов; легко тестировать и комбинировать.
- **Разделение API/реализации**: наружу экспортируется только контракт; внутри можно менять структуру состояния.
- **Сильнее тип-контроль**: `Robot` типизирован как отдельная сущность, которую нельзя “подменить” случайным объектом без ошибки.

### Пример использования

```python
from task_20.robot_adt import (
    WATER,
    Move,
    SetMode,
    Start,
    Stop,
    Turn,
    create,
    move,
    turn,
    set_mode,
    run,
    position,
)


# 1) Работаем с ADT через чистые функции
r0 = create(x=10, y=10, angle_deg=0, mode=WATER)
r1 = move(r0, 20)
r2 = turn(r1, 90)
r3 = set_mode(r2, "water")
print(position(r3))  # ок, но полей состояния нет


# 2) Исполняем программу как последовательность команд (task_15 style)
program = [Start(), Move(10), Turn(90), SetMode("soap"), Stop()]
r_final, log = run(program, r0)
print(position(r_final), log)
```

### Что важно для задания

- Мы **не раскрываем** внутреннее состояние робота: `Robot` opaque, а снаружи только чистые функции и `run()` как адаптер поверх \"старого\" backend.
- Можно добавлять новые операции/варианты интерпретации, не раскрывая внутренние детали и не превращая состояние в “общедоступный словарь”.
